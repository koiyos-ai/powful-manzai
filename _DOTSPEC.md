# _DOTSPEC.md ─ 1ドット認識プロトコル(パワフルマンザイ/pxv2)

**【確定 2026-07-20】** hero(下げ腕ベース)の全14,080ドット・24部位の認識をユーザー承認で確定。
被覆検査: 脱落0・重複0・幽霊claim 0。データ= `_pxmaster_hero.txt`(正準マップ)+ `_dots_*.txt`(24部位)。
以後この認識データベースが構造モデル第1層の正本。変更はユーザー裁定を経ること。

目的: `_pxv2_hero_base.png`(352×192)の全14,080不透明ドットを、部位ごとに
**1ドット単位で厳密に**帰属させる。「概ねこの範囲」という矩形・ゲート近似は禁止。

## 入力: マスターマップ `_pxmaster_hero.txt`
- 3行のコメント + `PALETTE:`(29行: `.`+28色) + `GRID:` + 192行のグリッド。
- **`GRID:` の直後の行が y=0**。以降1行が1つのy。行内の文字位置(0始まり)がそのままx。
  つまり「GRID:行のインデックス+1+y」行目・x文字目 = ドット(x,y)。
- 文字→色は PALETTE 節が正典(例: K=rgb(8,8,8), 1=rgb(24,24,24), S=rgb(248,200,168))。
- `.` は透過(ドットなし)。

## 出力①: ドットリスト `_dots_<part>.txt`(部位ごと)
```
# part: brow_L (左眉)
# 根拠メモがあれば書く
DOTS:
y=58: 152-153
y=59: 151-155
y=60: 149-157
AMBIGUOUS:
y=64: 154-164 @shared-with:eye_L 眉の底辺が目の上郭線と垂直連結
```
- `DOTS:` = その部位に確定帰属するドット。`y=NN:` にx範囲(`a-b`)と単独x を空白区切りで列挙。
- `AMBIGUOUS:` = 隣接部位と共有・境界が曖昧なドット。`@shared-with:<part>` と理由を必ず付ける。
- 部位名は必ず下のカタログの名前を使う。
- **_dots_*.txt の中身はASCIIのみ**(コメント・理由も英語/ローマ字で書く)。
  PS5.1がUTF-8日本語をCP932誤読して行が融合する実害があった(2026-07-16確認)。

## 出力②: ハイライト画像(自己検証用・必須)
```
powershell -ExecutionPolicy Bypass -File tools\px_mark.ps1 -dots _dots_<part>.txt -out _mark_<part>.png
```
- 緑=DOTS、橙=AMBIGUOUS で `_pxv2_hero_base.png` 上に重ねて描画される。
- **透過ドットを列挙すると ERROR が出る**(=存在しないドットを挙げた証拠。必ず直す)。
- 出力PNGを必ず Read して目視確認し、はみ出し・取り残しがあれば修正して再実行。

## 【重要ルール】輪郭線の共有(2026-07-17ユーザー指示)
**隣接するパーツの境界の輪郭線は、1本の線を両パーツで共有する。**
- 例: 「袖の輪郭線の最も手側」と「カフ(シャツ白帯)の輪郭線の最も肩側」は同一の線。
  「カフの最も手側」と「拳の最も肩側」も同一の線。
- 理由: 境目の輪郭線を最小化することがイラストとしての品質を高める(線を各パーツが
  別々に持つと境界が2重に太る)。
- データ表現: 共有線ドットは両パーツのファイルの `SHARED:` 節に記載し
  `@shared-with:<相手>` を付ける(所有の勝ち負けを決めない)。
- 線ドットのうち、片側のパーツの塗りにしか接しないものだけがそのパーツ単独の輪郭。

## 品質ルール(厳守)
1. ドットを帰属させる前に、必ずマスターマップの該当文字を実際に読むこと。
   読まずに「この範囲は全部○○のはず」と割り当てるのは禁止。
2. 全ドットはちょうど1部位に帰属するか、AMBIGUOUS として明示する。黙って重複・脱落させない。
3. 輪郭ドット(K等の暗色)は「それが縁取っている部位」に帰属。2部位の共有縁(袖と胴の縫い目等)は
   AMBIGUOUS で両部位に記載し `@shared-with:` で相互参照。
4. 曖昧さを隠さない。判断に迷うドットは AMBIGUOUS に置いて理由を書く方が正しい。
5. ゲーム本体のファイル(images/ 等)や既存pngを書き換えない。作ってよいのは
   `_dots_*.txt` / `_mark_*.png` のみ。gitコミットもしない。
6. 【絶対禁止・2026-07-18ユーザー指示】**境界線・輪郭線のトレースは、模様が複雑になった
   ("読み解きにくい")という理由だけで打ち切ってはならない。** 途中で止めて「以降は別の
   確定済みパーツが処理するはず」と憶測で確定させることは、実質的に未読の範囲を
   ルール1違反(読まずに割り当てる)で埋めているのと同じであり、禁止行為である。
   - トレース対象の線は、**それが実際に途切れる/他の確定済み構造物(カフ・フラップ等)の
     ドットと接触して統合される、その現物のドットまで**実測で追い切ること。
   - 「クラスタが増えて複雑になった」「模様が入り組んできた」は打ち切ってよい理由ではなく、
     むしろ**そここそ最も誤りやすい箇所なので追加の慎重さで読み進める**合図とする。
   - 一度でも「ここまでで十分だろう」という判断が頭をよぎったら、それは打ち切りの兆候。
     その時点で追跡を止めず、次の確定済み部位のドットに実際に到達するまで継続する。
   - 完了前セルフチェック: 「このトレースは実際の終端(他パーツとの接触点)まで到達したか、
     それとも自分の判断で範囲を区切ったか」を自問し、後者なら未完了として続行する。
   - 実例(2026-07-17): 袖/身頃の縫い目トレースをy=134で打ち切り、カフに接続する
     y=135〜138の3行を憶測で境界確定させた。実際は縫い目がその先も実在し
     (151→151→150→147と実測可能)、追跡すれば普通に見つかった。

## 【2026-07-17確立】輪郭線ファースト分解(ユーザー提案・服領域で実証済み)
服のように各部位が輪郭線で囲まれている領域は、ドット単位の目視帰属より
**輪郭線→閉領域→帰属**の順で機械的に分解する方が正確:
1. 線ネットワーク = 文字{K,1,2,/,-,G,g}のドット(X=56,24,24とB=88,40,24は靴の面や
   陰影クサビに使われるため「塗り」扱い。線はK/1/2//が本体)
2. 塗りドットの4近傍連結成分 = 閉領域(`tools/px_cc.ps1` → _cc_report/_cc_map/_cc_regions.png)
3. 領域→部位の対応表を人間が確定(38領域程度なのでドット単位より桁違いに誤りにくい)
4. 線ドットは「最寄りの塗り領域の部位」へ層同期BFSで帰属(同距離のみ優先度で決着。
   優先度を距離より強くすると前立てが線ネットワーク全体を侵食する——実害あり)
5. 幾何確定部(前立てx173-174/y128-150、ボタンx174-178/y135-138、フラップ縁)はBFS後に強制上書き
6. 縫い目が柔らかく領域が融合する箇所(袖⇔身頃)のみ、実測済みの縫い目で分割
実装: `tools/px_regen_body.ps1`(領域→18部位ファイルの自動生成、被覆0/0/0達成)

## 【2026-07-17追加】黒線でない部位(ボタン・フラップ)の認識=暗色階調ブロブ
ボタン・ポケットフラップは黒線でも暖赤の一本線でもなく、**その部位専用の暗い陰影階調**
(X=56,24,24・B=88,40,24・2=24,8,8・/=40,8,8)で描かれている。これはジャケット全体で
汎用される中間影(D/d/r)とは別クラスで、`tools/px_trace_feature.ps1 -blob` で実装:
1. 対象の暗色クラス{X,B,2,/}のみを候補にする(D/d/rを含めると前立てやラペル折り線などの
   汎用シェーディングまで暴走的に連結し、bbox全体を飲み込むことを実験で確認済み)
2. 目視確認済みの種(シード)座標から8連結でフラッドフィル
3. 結果を検証画像で目視確認(ボタン=リング、フラップ=フック形状として正しく分離)
初回の「ボタン=矩形」「フラップ=x/y窓の使い回し」「身頃=残差」は全て範囲近似の再発であり、
この階調ブロブ法で作り直した。身頃はボタン/フラップ/前立て/ラペル/袖/襟を正しく引いた後の
残りとして再構成され、共有輪郭線ルールにより境界(V字折り線・肩線・縫い目・裾)は
隣接パーツと相互参照される(独立した染み出しがなくなった)。
