## 0. セッション運用プロトコル(全チケット共通・厳守)

### 0.1 セッションの始め方
1. `ROADMAP.md` で現在フェーズと完了済みチェックを確認。
2. 本ファイルの該当チケットを読む。「参照」欄の設計mdの該当節を**必ず読んでから**着手。
3. 着手前に `git status` がクリーンであることを確認(散らかっていたら先に報告)。

### 0.2 作業の掟
- **設計→md反映→実装**の順。仕様に迷ったら勝手に決めず、設計mdを確認→無ければユーザーに質問。
- UIは `DESIGN.md` 準拠(100dvh・角丸四角・色トークン・ドット文字の使い分け)。
- キャラ絵は `tools/PXART_PROCESS.md` 完全準拠。**生成AI(fal.ai等)は使わない。鼻と口は絶対に描かない。**
- 画面間の本番往復プロトコル(`ARCHITECTURE §2`)を壊さない。localStorageスキーマ変更は必ず `migrate()` 追加+`ARCHITECTURE §3` 追記とセット。
- 得点計算の配置原則(`ARCHITECTURE §5`): 舞台の上=manzai-play、舞台の外=success-play。同じ式を両側に書かない。
- 一時ファイルは `_` 接頭辞(gitignore済)。使い捨て確認HTMLは役目を終えたら `_archive/` へ。

### 0.3 技術的な注意(ハマりどころ)
- HTML/JSはUTF-8。**PowerShellの`Get-Content`で日本語が化ける**ので、ファイル内容はReadツールで読むこと。
- 巨大HTML(success-play 55KB等)は全読みせず、Grepで関数名を当ててから該当行をReadする。
- Windows PowerShell 5.1(`&&`不可・ternary不可)。POSIXが要るならBashツール。
- mp3をfetchするため `file://` 直開き不可。検証は静的サーバ経由(下記0.4)。
- iOS Safariのオーディオは初回タップ解禁が必要 → 既存の `ensureAudio()` パターンを踏襲。

### 0.4 検証と完了処理(全チケット共通のDoD付帯条件)
1. ローカル静的サーバ(ポート8090慣例)で対象画面を開き、コンソールエラーゼロを確認。
2. 実機確認が要るもの(UI/演出/操作)は cloudflared トンネルURLをユーザーに案内して確認を得る。
3. 既存フローの回帰確認: 「新規開始→コマンド1回→劇場出番→漫才プレイ→帰還反映」が壊れていないこと。
4. コミット(日本語・1行要約+本文)。`ROADMAP.md` の該当チェックボックスを更新して同コミットに含める。

## P1-01 セーブスキーマv2とマイグレーション基盤
- **目的**: 以後のチケットが安全にフィールドを足せる土台。
- **参照**: `ARCHITECTURE §3.3`
- **触る**: `success-play.html`
- **手順**:
  1. `S` 初期化(`newGame`)に `v:2, year(=yearOf(turn)キャッシュ不要なら省略), tokusyu:[]` を追加。
  2. `loadGame()` 冒頭に `migrate(save)`: `v`無し→v2へ(欠損フィールドにデフォルト付与)。以後 `v===SAVE_V` まで順送りする switch 構造にする。
  3. 新キー `pm_flags`(`{seen:{},vars:{}}`)と `pm_meta`(`{endings:{},combos:{},runs:0}`)のload/save関数を追加。**`clearSave()` は `pm_meta` を消さない**。`newGame()` は `pm_flags` をリセットし `pm_meta.runs++`。
- **受け入れ**: 旧セーブ(手でv抜きJSONを入れて)を読んでも壊れない。新規開始→リロードで継続できる。`clearSave`後も`pm_meta`が残る。
- **検証**: DevToolsのApplication→localStorageで3キーの中身を目視。

# Phase 8. エンジン実装(完全版 v2.0)

> 依存順: P8-01 → P8-02(**最大リスクを最初に**) → P8-03 → P8-04〜06 → P8-07 → P8-08 → P8-09
> 新規リポジトリ(または `engine/` サブディレクトリ)で作業。方式はP8-01でユーザーと決める。

## P8-01 プロジェクト雛形+書き出し検証
- **手順**: 選定エンジンで空プロジェクト作成→解像度/向き(縦)/ピクセルアート設定(テクスチャフィルタ無効)→iOS/Android/PC の**空ビルドが通る**ことを最初に確認(署名・SDK等の環境問題を先に潰す)。リポジトリ方式(新規repo/モノレポ)をユーザーに確認。
- **受け入れ**: 3プラットフォームのビルド成果物が手元で起動する。

## P8-02 リズムゲーム技術検証スパイク
- **目的**: 「オーディオクロックに完全同期した判定」がエンジンで成立するかの検証。**ここが全計画の最大リスク。**
- **手順**: 検証専用シーン: BPM150のクリック音+スクロールノート+タップ判定窓(HTML版と同値)を実装し、実機(iPhone/Android)で入力遅延を計測(判定オフセット調整UI付き)。Godotなら `AudioServer.get_time_since_last_mix()` 系の補正を使用。満足な精度が出なければ**P7-01へ差し戻してエンジン再検討**。
- **受け入れ**: 実機でHTML版と同等以上の判定気持ちよさ(ユーザー体感承認)。計測値をmdに記録。

## P8-03 コア移植: データ層+シナリオ/イベントエンジン
- **手順**: `data/*.json` 読込→`evalCond`/`applyEff` 移植→テストベクタ(P7-03)のうち該当分を通す→紙芝居UI(立ち絵/文字送り/選択肢)をエンジンUIで再実装→TEST-01台本とS-00で動作確認。
- **受け入れ**: S-00がHTML版と同一の台詞・分岐で再生される。

## P8-04 育成画面移植
- **手順**: 状態管理(セーブ=P7-05スキーマ)→カレンダー/コマンド/ガチャ/成長/劇場出番/M-1進行→テストベクタ(ガチャ/成長/得点換算)通過。UIはDESIGN.mdのトークンをエンジンのテーマとして再構築。
- **受け入れ**: 3年通しプレイがエンジン版単体で完走できる。

## P8-05 漫才プレイ本実装 / P8-06 ネタビルダー+タイトル移植
- **手順**: P8-02のスパイクを製品化(譜面生成=JSONタイミング定義から/温度/タグ属性/カットイン)→ネタビルダー→タイトル画面→画面間遷移(エンジンのシーン遷移)。
- **受け入れ**: 全画面がエンジン版で揃い、本番往復(育成⇄漫才)が成立する。

## P8-07 パリティQA
- **参照**: `ARCHITECTURE §9.2`(パリティ定義)
- **手順**: ①全テストベクタ通過 ②同一操作の並走プレイ(HTML版とエンジン版で同じ選択をして結果を照合。乱数はシード固定のデバッグモードを両版に用意) ③P5-02の4シナリオQAをエンジン版で再実施。
- **受け入れ**: 仕様差ゼロ(意図的な演出差は除き、mdに列挙)。

## P8-08 エンジン版演出強化
- **手順**: HTML版で我慢していた演出をユーザーとブレストして追加(パーティクル/画面振動/触覚フィードバック/リッチなトランジション等)。**仕様(数値・判定)は変えない**。
- **受け入れ**: ユーザーが「完全版の顔」に納得する。

## P8-09 ストア配信準備 → v2.0
- **手順**: アイコン/スプラッシュ/スクリーンショット→プライバシーポリシー等の審査要件→開発者アカウント(Apple $99/年、Google $25 初回のみ※要ユーザー契約)→内部テスト配信(TestFlight/内部テスト)→公開判断はユーザー。
- **受け入れ**: ユーザーの実機にストア経由(またはテスト配信)でインストールでき、`v2.0` タグが打たれている。
